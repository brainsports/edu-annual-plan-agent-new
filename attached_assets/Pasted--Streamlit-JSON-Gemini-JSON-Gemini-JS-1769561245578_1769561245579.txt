목표: Streamlit 앱에서 ‘JSON 파싱 실패(Gemini 응답이 올바른 JSON 객체가 아님)’ 오류를 근본 해결한다.
원인: Gemini 응답이 중간에서 끊기거나(JSON 미완성), 호출 위치별로 generation_config가 달라 출력이 길어져 JSON이 완성되지 못한다. 또한 간헐적으로 코드펜스/개행문자 등으로 JSON 규격이 깨진다.

수정 범위: utils.py, doc_utils.py, main.py (Gemini 호출/파싱/버튼 로직이 있는 곳 전부)

작업 1) generation_config 단일화(중복 제거)
- 프로젝트 전체에서 genai.GenerationConfig(...)를 검색해 중복 정의가 있으면 제거한다.
- utils.py에 DEFAULT_GEN_CONFIG를 하나만 만든다:
  - response_mime_type="application/json"
  - temperature=0.2~0.7 (현 설정 유지 가능)
  - max_output_tokens는 가능한 큰 값으로(현재 8192 유지)
- 모든 Gemini 호출은 반드시 utils.py의 DEFAULT_GEN_CONFIG를 import해서 사용하도록 통일한다.
- 호출 코드에서 개별적으로 config를 다시 덮어쓰지 않게 한다.

작업 2) “서버에서 먼저 검증” + 자동 재시도/분할 생성
- utils.py에 safe_gemini_json(prompt) 함수를 만든다.
  요구사항:
  1) model.generate_content(...) 실행 후 response.text를 얻는다.
  2) 1차로 json.loads(raw_text) 시도
  3) 실패 시 raw_text에서 코드펜스(``` ... ```) 제거하고,
     첫 번째 '{'부터 마지막 '}'까지 잘라 재시도한다.
  4) 그래도 실패하면 “응답이 끊긴 것(닫는 따옴표/중괄호 누락)”으로 판단하고,
     같은 내용을 더 짧게 만들도록 지시한 ‘재요청 프롬프트’를 1~2회 자동 실행한다.
     - 재요청 프롬프트 규칙:
       a) ‘오직 JSON만 출력’
       b) 모든 문자열 줄바꿈은 반드시 \\n으로 표현(실제 개행 금지)
       c) 각 필드는 700자 이내로 요약(끊김 방지)
       d) 마크다운/설명 문장/코드펜스 금지
  5) 최종 실패 시 예외를 올리고, main.py에서 st.error로 원문을 st.code로 보여준다(디버깅용).

작업 3) “필드 단위 분할 생성”으로 구조 개선(끊김 근본 예방)
- doc_utils.py에서 part1_general처럼 긴 텍스트를 한 번에 큰 JSON으로 생성하는 함수를 찾는다.
- 아래 방식으로 바꾼다:
  - need_1_user_desire, need_2_1_regional 등 ‘각 필드’를 개별 호출로 생성한다(짧은 출력).
  - 각 호출은 utils.safe_gemini_json 또는 utils.safe_gemini_text(문자열 전용)로 받아온다.
  - 마지막에 Python dict로 조립해서 완성 JSON을 만든다.
  - 저장/세션 상태에도 조립된 dict를 넣는다.
- 이렇게 하면 모델 출력이 길어져 JSON이 끊기는 문제가 거의 사라진다.

작업 4) 프롬프트 강제(모든 호출 공통)
- Gemini에게 전달하는 프롬프트 템플릿 상단에 아래 규칙을 공통으로 삽입한다:
  - “반드시 단일 JSON 객체만 출력”
  - “마크다운 금지”
  - “문자열 줄바꿈은 \\n으로만”
  - “키 이름은 지정된 스키마 그대로”
  - “마지막은 반드시 } 로 끝나야 함”
- 샘플 데이터 로드/AI 분석 시작 버튼 흐름을 그대로 유지하되,
  JSON 생성 부분만 위 유틸을 거치게 한다.

검증 기준(테스트)
1) ‘AI 분석 시작’ 클릭 시 더 이상 JSON 파싱 실패가 뜨지 않는다.
2) 응답이 길어져도(표/필드가 많아도) 필드 분할 생성으로 끊김이 재발하지 않는다.
3) 실패하더라도 자동 재요청 후 성공하거나, 최종 실패 시 원문 출력으로 원인 확인이 가능하다.

주의
- 프론트(스트림릿)에서 JSON.parse 같은 이중 파싱이 있다면 제거하고,
  서버에서 dict로 만든 뒤 st.session_state에 dict로 저장하도록 정리한다.
