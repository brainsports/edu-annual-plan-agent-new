[문제 요약]
1) PART 1은 화면(폼)에서는 내용이 전부 잘 채워지는데, Word 다운로드 결과가 PART 1의 “일부분만” 나오거나(혹은 텍스트만 단순 출력)해서, 내가 기대하는 “PART 1 전체 폼 구성”대로 다운로드가 되지 않아.
   - PART 1 다운로드는 ‘PART 1에 해당하는 모든 섹션/테이블/만족도 결과’까지 포함되어야 해.
   - 단, “전체 다운로드” 버튼은 절대 만들지 말아줘. 다운로드는 반드시 PART 1~PART 4 각 버튼으로만 분리되어야 해.

2) 만족도(설문) 결과가 업로드 데이터에서는 샘플데이터처럼(표 + 차트/분포) 제대로 나오지 않아.
   - 샘플데이터에서는 ‘문항별 응답 분포 표 + 평균/분포 차트 + 전체 평균’이 잘 나오는데, 업로드 데이터에서는 일부 요약만 나오거나 글자수 규칙을 못 맞추는 등 결과 품질이 떨어져.
   - 원인을 정확히 찾아서 “샘플데이터와 동일한 파이프라인/표현”으로 나오게 해줘.

[내가 추정하는 원인(가설)]
A) PART 1 Word 생성 함수가 UI에 있는 전체 state(섹션/테이블/만족도 결과)를 다 읽지 못하고, 일부 키만 가져오거나 중간에 누락되는 것 같아.
   - 예: generate_part1_report()가 part1 딕셔너리 전체를 순회하지 않거나, 키 매핑이 일부만 연결되어 있을 수 있어.
   - 혹은 textarea 값은 session_state에 있는데 docx 생성 시점에 다른 변수(초기값/샘플값/빈 값)를 참조하는 것 같아.

B) 만족도 결과는 “샘플 데이터 로더”에서는 정규화된 구조(질문 리스트/점수 분포/응답자수)가 이미 맞춰져 있는데,
   업로드 데이터에서는 분류 결과 → 만족도 스키마 변환 단계가 빠졌거나, 컬럼명이 달라서 차트용 데이터프레임 생성이 실패하는 것 같아.

[요구사항(반드시 지켜줘)]
1) ‘전체 다운로드’ 버튼을 새로 만들지 말 것.
   - PART 1, PART 2, PART 3, PART 4 각각의 다운로드 버튼만 유지.
   - 각 버튼은 해당 파트만 docx로 생성/다운로드.

2) PART 1 다운로드는 “PART 1 화면에 표시되는 전체 내용”이 빠짐없이 들어가야 함.
   - 포함 대상(최소):
     - 사업의 필요성: need_1_user_desire, need_2_1_regional, need_2_2_environment, need_2_3_educational
     - 전년도 사업평가 및 환류계획: feedback_table, total_review_table
     - 만족도조사: (표 형태의 문항별 결과 + 총 응답인원 + 전체 평균) + 주관식 요약 + 종합 분석/제언(각 500자 규칙)
     - 사업목적: purpose_text
     - 사업목표: goals_text(불릿 5개 규칙)
   - “화면에서 보이는 순서”와 동일한 순서로 문서에 배치.
   - 표는 Word에서도 표로 유지(최소한 table 형태로 출력). 텍스트만 이어붙인 형태로 망가지면 안 됨.

3) 만족도 결과(업로드 데이터)도 샘플데이터와 동일한 UI 결과가 나오게 해줘.
   - 문항별 응답 분포 표(5점~1점) + 문항별 평균 + 전체 평균(예: 4.13점) 표시
   - 평균값 막대(가로) / 응답 분포(스택) 같은 차트가 샘플처럼 렌더링
   - 주관식 문항 분석(500자 이상) / 종합 분석 및 제언(500자 이상)도 규칙 만족

[구체 작업 지시(어디를 보고 무엇을 고칠지)]
- doc_utils.py / utils.py / main.py(또는 streamlit 페이지)에서 아래를 순서대로 점검하고 수정해줘.

(1) PART 1 docx 생성 흐름 점검
- PART 1 다운로드 버튼이 호출하는 함수(예: generate_part1_report, export_part1_docx 등)를 찾아서,
  입력 데이터가 “현재 화면에 들어있는 part1 결과 전체”인지 확인해줘.
- 만약 part1 데이터가 session_state에 저장되어 있다면, docx 생성 함수는 반드시 그 session_state의 최신 값을 사용하게 해줘.
- PART 1 문서에 포함되어야 하는 섹션 키들을 명시적으로 매핑해서, 누락 없이 출력되게 해줘.
  예) PART1_KEYS = [need_1_user_desire, need_2_1_regional, need_2_2_environment, need_2_3_educational, feedback_table, total_review_table, satisfaction_summary, subjective_analysis, overall_suggestion, purpose_text, goals_text]
  (실제 키 이름은 프로젝트에 맞게 조정)
- “한 섹션만 나오고 끝나는” 문제가 있으면:
  - return/exit가 중간에 있는지,
  - 예외 발생 후 fallback으로 빈 문서가 저장되는지,
  - 또는 리스트/딕셔너리 순회가 중간에 끊기는지 확인해줘.

(2) 만족도 파이프라인 점검(샘플 vs 업로드 동일화)
- 샘플데이터 로드 함수가 만들어내는 만족도 구조(질문 목록, 분포 카운트, 평균, 응답자수)와
  업로드 데이터 분류 결과가 만들어내는 구조를 비교해줘.
- 업로드 데이터에서도 동일한 “정규화 함수(normalize_survey_data 같은)”를 거쳐서
  차트/표에 들어가는 데이터프레임 컬럼이 100% 동일해지게 해줘.
- 질문 텍스트/점수 컬럼명이 다를 경우에도 자동으로 매핑되도록 보정 로직을 넣어줘.
  (예: ‘5점(명)’, ‘5점’, ‘5’ 같은 표기를 모두 5로 통일)
- 차트 렌더링이 실패하면 사용자 화면에 조용히 빈칸이 되지 말고,
  디버그 expander에 “어느 단계에서 어떤 컬럼이 없어서 실패했는지”를 1줄로 표시해줘(기본은 접힘).

(3) 글자수/불릿 규칙(이미 성공한 부분 유지 + 표/만족도 실패를 완전히 해결)
- 표 셀(problem/improvement/content 등)은 “글자수(min/max) + bullet_count” 규칙을 반드시 만족하도록,
  LLM 출력 후에 후처리로:
  - bullet_count가 부족하면 의미를 해치지 않는 범위에서 항목을 쪼개서 불릿 수를 맞추고
  - min_chars_no_space가 부족하면 ‘중복 문장’이 아니라 “동일 의미의 구체화 문장”을 추가해서 자연스럽게 채워줘
  - max 초과 시에는 불릿 우선 유지하면서 문장 압축(truncate)해줘
- 특히 만족도 ‘주관식 문항 분석(500자 이상)’과 ‘종합 분석 및 제언(500자 이상)’이 업로드 데이터에서도 항상 통과하게 보장해줘.

[완료 기준(내가 확인할 체크리스트)]
1) UI에서 업로드 데이터로 분석 → PART 1~4 각각 다운로드 가능(전체 다운로드 없음)
2) PART 1 다운로드 docx에 PART 1의 모든 섹션과 표(환류계획/총평/만족도 표)가 빠짐없이 포함
3) 만족도 결과는 샘플데이터와 동일하게 표 + 차트 + 평균점수까지 화면에 표시
4) 규칙 검증 로그에서 표 셀/주관식/제언 모두 글자수 조건 통과

이 조건을 만족시키는 더 좋은 설계가 있으면 그걸로 해줘. 없다면 위 요청대로 정확히 작업해줘.
