문제를 해결하려고 하는데 내 생각은 아래와 같아. 네가 더 좋은 대안이 있으면 그렇게 하고, 없으면 요청대로 작업해줘.

[1) 크래시 오류(최우선) — 반드시 먼저 해결]
- 현재 앱이 워드 생성 과정에서 바로 크래시가 난다.
- 에러 로그:
  File "main.py", line 622: part1_report = generate_part1_report(...)
  File "doc_utils.py", line 312: set_table_width_by_ratio(table, [0.20, 0.40, 0.40])
  File "doc_utils.py", line 107: tblPr = tbl.get_or_add_tblPr()
  AttributeError: 'CT_Tbl' object has no attribute 'get_or_add_tblPr'

[오류 원인]
- python-docx 버전에 따라 CT_Tbl에 get_or_add_tblPr() 메서드가 없어서 발생한다.
- 즉, doc_utils.py의 set_table_width_by_ratio 구현이 현재 런타임의 python-docx와 호환되지 않는다.

[오류 수정 목표(완료 기준)]
- 샘플 데이터/실제 업로드 모두에서 워드 생성이 크래시 없이 끝난다.
- 표 폭 비율 고정 기능(20/40/40 등)은 유지된다.
- python-docx 버전 차이에도 동작하는 “호환형 구현”으로 교체한다.

[오류 수정 지침(구체)]
A) doc_utils.py의 set_table_width_by_ratio()에서
- tbl = table._tbl(CT_Tbl)에 대해 tbl.get_or_add_tblPr()를 절대 호출하지 말아라.
- 대신 아래처럼 tblPr를 안전하게 가져오거나 생성하는 방식으로 바꿔라:
  1) tblPr = tbl.tblPr 를 먼저 시도
  2) tblPr가 None이면 OxmlElement('w:tblPr')로 생성 후 tbl에 insert(0, tblPr)
- 표 레이아웃을 고정으로 설정한다(자동폭 방지):
  - table.autofit = False
  - tblPr 아래에 w:tblLayout w:type="fixed"를 추가(이미 있으면 재사용)

B) 컬럼 폭 적용 방식(비율 → 실제 폭)
- 페이지 usable_width(페이지 폭 - 좌/우 마진)를 계산해서, ratios에 따라 실제 폭을 만든다.
- 폭 적용은 “컬럼/셀 단위로 반복 적용”한다(가장 안정적):
  - 모든 행의 각 셀(cell.width)에 폭을 반복 지정
  - 가능하면 table.columns[i].width도 함께 지정(지원 시)
- usable_width 계산이 실패해도 크래시 나지 않게 기본값으로 fallback 한다.

C) 예외/로그
- set_table_width_by_ratio에서 예외가 나면:
  - 어떤 표/어떤 ratios에서 실패했는지 로그를 남기고
  - 문서 생성이 전체 중단되지 않게(가능하면) 안전하게 처리한다.

[2) PART 2 세부 사업 계획 수정 — 작성지침 + 워드 표 폭 비율]
(오류 수정 후 아래를 진행)

[PART2 핵심 문제 2가지]
1) 작성지침(표2) 미준수
- 기대효과가 너무 길다 → “공백 포함 100자 이내”로 강제해야 한다.
- 계획내용이 긴 서술/반복문장 형태다 → 사무용 보고서 형식으로 “3~5개 항목”만 간단히 써야 한다.
- 계획내용은 수치/시간/횟수/빈도/인원 등 구체 정보가 반드시 포함되어야 한다.
- “(추가 내용 필요)” 같은 문구가 남으면 안 된다.
- 의미 없는 공통문장(아동의 건강한 성장… 반복)은 금지한다.

2) 워드 출력 Part2 표 가로폭(컬럼 폭) 비율 불일치
- Part2 세부사업내용 표(7열) 비율을 아래처럼 고정:
  [세부영역, 프로그램명, 기대효과, 대상, 인원, 주기, 계획내용]
  - 기대효과 30%
  - 계획내용 30%
  - 나머지 5개 컬럼 합 40%를 균등 분배(각 8%)
  => 최종 비율: 8% / 8% / 30% / 8% / 8% / 8% / 30%
- 모든 세부영역(보호/교육/문화/정서/지역연계 등) 표에 동일 적용한다.

[PART2 수정 목표(완료 기준)]
A) 기대효과
- 100자 이내(초과 금지), 1~2문장, 핵심만

B) 계획내용
- 3~5개 불릿(항목)만 생성/표시/출력
- 각 항목은 짧고, 최소 1개 이상의 구체 정보 포함(숫자/시간/횟수/빈도/인원/담당 등)
- “(추가 내용 필요)” 0건
- 포괄/반복 문장 0건

C) 워드 표 폭
- Part2 표에서 비율(8/8/30/8/8/8/30)이 줄바꿈이 있어도 유지된다.
- 자동폭(autofit) 때문에 비율이 깨지지 않는다.

[PART2 구현 지침]
1) 생성 단계(콘텐츠 규칙 강제)
- 기대효과 생성 직후 100자 초과 검사:
  - 초과 시 자동 축약 또는 재생성(안전장치)
- 계획내용은 문자열 1덩어리로 만들지 말고,
  - 내부적으로 planItems: string[] 구조(길이 3~5)로 관리한 뒤
  - UI/Word에서 불릿 목록으로 렌더링한다.
- 계획내용 생성 규칙:
  - 항목 수 3~5 고정(2개 이하/6개 이상 금지)
  - 각 항목에 구체 정보 최소 1개 포함
  - “(추가 내용 필요)” 절대 금지
  - “건강한 성장/안전한 환경…” 같은 포괄문 반복 금지

2) UI(미리보기 표)
- Part2 표에서 기대효과 100자 제한을 그대로 반영한다.
- 계획내용은 3~5개 불릿으로 표시한다(긴 서술 1덩어리 금지).
- 가능하면 UI 표 컬럼도 동일 비율(8/8/30/8/8/8/30)을 적용한다.

3) 워드(docx) 출력
- Part2 세부사업내용 표 생성 시,
  - set_table_width_by_ratio(오류 수정된 호환형 버전)를 호출하여
  - ratios=[0.08,0.08,0.30,0.08,0.08,0.08,0.30] 적용
- 셀 패딩/들여쓰기 때문에 폭이 틀어지면 가능한 범위에서 최소화한다.
- 계획내용 셀은 planItems 배열을 불릿으로 출력(3~5개만).
